## 1. КОНКУРЕНТЫ

Отобраны 4 севастопольских ресторана близкого тира + 1 эталонный мировой пример (для best practices). Главные критерии — географическая близость (центр Севастополя), сопоставимый средний чек, наличие лендинга.

### 1.1. Мекензи 1783 — самый близкий конкурент
**URL:** [mekenzi.ru](https://mekenzi.ru/)
**Что это:** Панорамный ресторан на площади Нахимова (то есть **через дорогу от MOMO**). Прямой географический конкурент.

| Что есть | Чего нет |
|---|---|
| **3D-тур по залу** (360° панорама) — сильная фишка | Нет встроенного чата |
| Программа лояльности через Loona | Онлайн-бронирование сводится к «позвонить» |
| Календарь событий | На главной мало info о меню |
| Раздел «Атмосфера» с командой | Скучноватый минимализм без характера |

**Что брать:** идея 3D-тура мощная — для туриста, который выбирает «у кого вид лучше», это решающий аргумент. Можно сделать VR-превью террасы MOMO.
**Что НЕ повторять:** «позвоните по будням 10–18» — отрезает вечерних гостей и туристов вне рабочего дня. У нас рядом со звонком — резервная форма «перезвоните мне» с SLA 15 минут.

### 1.2. На Юга — конкурент по «новой крымской»
**URL:** [nayuga-sev.ru](https://nayuga-sev.ru/)
**Что это:** Ресто-бар «новой крымской кухни», в одной экосистеме с сетью Samurai. Похожая идея «локальные продукты, авторская подача».

| Плюсы | Минусы |
|---|---|
| Большой каталог блюд с фото и ценами | Сайт сделан как **доставка**, не как лендинг ресторана |
| Корзина, онлайн-заказ | **Нет** кнопки бронирования стола |
| Афиша мероприятий, отзывы | Темная тема, тяжёлая, перегружена карточками |
| Интеграция с сетью (Samurai) | Слабый storytelling — нет «почему именно сюда» |

**Что брать:** разделение «доставка» и «зал» через ясный UX, наличие афиши событий.
**Что не повторять:** их сайт не отвечает на вопрос «что я получу, если приду в зал». MOMO должен это решать.

### 1.3. Сосновый Бор — семейная аудитория, природа
**URL:** [restbor.com](https://restbor.com/)
**Что это:** Ресторан в сосновом лесу за городом, основной фокус — банкеты, свадьбы, корпоративы, семейный отдых.

| Плюсы | Минусы |
|---|---|
| **Бонусная карта** с QR-кодом — сильный инструмент удержания | **Нет фотогалереи интерьера** — критичный пробел для премиум-сегмента |
| 6 чётких блоков «банкет / свадьба / тур-ланч / мастер-класс» | Нет видео, нет атмосферы |
| Форма обратного звонка | Карусель меню — устаревший паттерн на 2026 |
| Интеграция с ЮMoney для предоплат | Сделан на Tilda — заметно по типовым блокам |

**Что брать:** концепция бонусной карты + блочное разделение услуг (банкет / тур-ланч / детский праздник).
**Что не повторять:** отсутствие визуала интерьера. У MOMO интерьер и вид на бухту — главная ценность, прятать его — преступление.

### 1.4. Гастрономика — банкетный пример
**URL:** [gastronomika-rest.ru](https://gastronomika-rest.ru/)
**Что это:** Ресторанный комплекс с 5 банкетными площадками, заточен под мероприятия.

| Плюсы | Минусы |
|---|---|
| Чёткая навигация: 5 площадок × 4 типа событий | Не сайт *ресторана*, а сайт *банкетного комплекса* |
| Портфолио проведённых мероприятий | Нет повседневного меню |
| Каналы связи: VK, Telegram, ЯКарты | Скучный «корпоративный» вид |
| Кнопка «Узнать свободные даты» — умный CTA | Слабая визуальная иерархия |

**Что брать:** CTA «Узнать свободные даты» — конкретный, не размытый.
**Что не повторять:** для MOMO банкеты — побочная функция, не делать их главным сюжетом.

### 1.5. White Rabbit Moscow — мировой эталон
**URL:** [whiterabbitmoscow.com](https://whiterabbitmoscow.com/en/)
**Что это:** Ресторан #1 в России, World's 50 Best, шеф Владимир Мухин. Не прямой конкурент, но эталон того, как делать сайт премиум-ресторана.

| Что брать | Контекст |
|---|---|
| **3D-тур + выбор конкретного стола онлайн** | Гость может выбрать стол у окна, как место в кино |
| Депозит при бронировании (10 000 ₽/чел.) | Снижает no-show — гости, которые внесли деньги, приходят |
| Storytelling шеф-повара и кухни | На лендинге не «у нас есть еда», а «у нас есть автор» |
| Подписка на новости (приватные дегустации) | Лидген: email-база лояльных гостей |
| Английская версия | Туристы. Для MOMO — релевантно |

**Что не повторять:** депозит 10К — для среднего чека MOMO (~1000 ₽) это перебор. Но **символический депозит 500 ₽** или **залог карты** на большие столы по выходным — рабочая механика для снижения no-show.

### 1.7. Главные выводы по конкурентам
1. **3D-тур / превью террасы и зала** — отличает 2 лучших сайта (Мекензи и White Rabbit) от остальных. Сильный аргумент для туриста.
2. **Storytelling кухни и шефа** — никто из локальных не делает. Свободная ниша для MOMO.
3. **Депозит на бронь** — никто из локальных не использует. Можно стать первыми в Севастополе и снизить no-show.
4. **Программа лояльности** (digital-карта в любом канале — SMS/email/QR) — никто из локальных не реализовал. Свободная ниша на V3.
5. **Английская версия** — есть только у White Rabbit. Для Севастополя с большим потоком туристов — упущенная ниша.

### 2.4. Что НЕ делать (важно)
Ниже — функции, которые часто появляются «по инерции», но для MOMO нерелевантны:
- ❌ **Онлайн-заказ доставки** — у momokitchen уже есть отдельный контур.
- ❌ **Корзина / онлайн-оплата** — для лендинга-ресторана это «лишняя дверь».
- ❌ **Регистрация / личный кабинет** — для бронирования достаточно звонка.
- ❌ **Чат-бот на сайте** — добавляет ещё одну очередь коммуникации поверх телефона. На MVP вреден.
- ❌ **Telegram-бот бронирования** — отложен в V2/V3. На старте звонок надёжнее (см. project.md §7.5).
- ❌ **Тёмная тема** — конфликтует с фирменным светлым стилем.

## 3. UX И СЦЕНАРИИ

Три типичных сценария взаимодействия с лендингом ресторана. Для каждого — пользователь, путь, что удобно, что бесит, как мы это решаем.

### 3.1. Сценарий А: «Турист в Севастополе на вечер»

**Кто:** Алексей, 38 лет, приехал с женой и сыном (8 лет) на 3 дня. Сейчас 17:30, гуляют по Приморскому бульвару, ищут «где красиво поесть с видом на море».

**Путь:**
1. Гугл/Яндекс → «рестораны Севастополь с видом на море семейные»
2. Открывает 3–4 сайта в фоновых вкладках
3. На каждом тратит 15–30 секунд
4. Выбирает 1, бронирует или просто «доходит и спрашивает»

**Что удобно:** один-два кадра с видом + цена/средний чек **на первом экране**. Кнопка «Маршрут» без скролла. Понятно, есть ли свободно сейчас.

**Что бесит:** видео-автоплей со звуком, тяжёлая загрузка на 4G, всплывающие cookie-окна на 80% экрана, обязательная регистрация для бронирования, неработающая мобильная вёрстка.

**Решение для MOMO:**
- Hero — статичная фотография (а не видео), вес < 250 KB, LCP < 2.5s на 4G.
- На первом экране: вид на бухту + средний чек + «детская комната» + кнопка «Бронь стола» и «Маршрут».
- Мобильная версия — приоритет, не «как-нибудь».
- Cookie-баннер — нижняя тонкая полоса, без блокировки контента.

### 3.2. Сценарий Б: «Местный планирует свидание»

**Кто:** Мария, 29 лет, живёт в Севастополе. В пятницу годовщина отношений. Сейчас вторник вечером.

**Путь:**
1. Открывает Instagram MOMO, листает посты — атмосферу знает.
2. Идёт на сайт **за конкретикой**: меню, бронь стола у окна, цены.
3. Хочет забронировать на пятницу 19:30, стол у окна, заранее.
4. Если бронь сложная — закроет вкладку и напишет в директ Instagram (что для ресторана хуже — потеряется в потоке).

**Что удобно:** возможность выбрать **конкретное место** (стол у окна), указать повод (годовщина — может, подарят что-то), мгновенное подтверждение, напоминание за день.

**Что бесит:** «отправьте заявку, мы перезвоним в течение дня» — нет, она хочет результата сейчас. Длинная форма с 8 полями. Отсутствие выбора стола.

**Решение для MOMO:**
- Звонок: один тап → разговор с администратором → дата/время/гости/повод собираются голосом за 1–2 минуты. Тёплый контакт, человек запоминает атмосферу заведения с первого звонка.
- Если ресторан занят / нерабочие часы — форма «Перезвоните мне»: имя, телефон, окно «когда удобно». Обратный звонок в течение 15 минут в рабочее время.
- Поле «когда удобно» в форме позволяет администратору не дёргать гостя в неудобный момент.

### 3.3. Сценарий В: «Семья выбирает воскресный обед»

**Кто:** Игорь, 42 года, жена и двое детей (3 и 7 лет). Воскресенье 11:00, ребёнок 3 лет проснулся в 6:30, всем уже хочется сбежать из дома.

**Путь:**
1. Спрашивает жену: «куда?», она: «куда-то где детям нормально».
2. Гуглит «семейные рестораны Севастополь детская комната».
3. Открывает несколько сайтов, ищет **подтверждение** что «нам там будут рады с детьми».
4. Хочет позвонить и убедиться вживую.

**Что удобно:** **фото детского уголка**, реальные дети в кадре (или хотя бы детская порция), отдельный блок «для семей», явно прописанные «стульчики / пелёнки / раскраски». Кнопка «Позвонить» большая, в один тап.

**Что бесит:** сайт с пафосом fine-dining без намёка на детей, фото взрослых на свидании — отталкивает («нам там не место»). Невозможно понять, есть ли детское меню без скачивания PDF.

**Решение для MOMO:**
- Отдельная секция «Маленьким гостям» с **реальным фото детского уголка и детской порции** (на бирюзовом фоне — повторяет фирстиль).
- Чек-лист «у нас удобно»: 4 пункта с иконками (стульчики, пелёнки, мультики, мастер-классы).
- Кнопка «Позвонить» — крупная, рядом с CTA «Забронировать» (для тех, кому нужно убедиться голосом).
- В форме «Перезвоните мне» — поле «комментарий» с подсказкой «детский стул, коляска, день рождения и т.д.».

### 3.4. Общие UX-паттерны (что считается базовой гигиеной)

| Паттерн | Зачем |
|---|---|
| Sticky-header с кнопкой «Бронь» | На длинной странице CTA всегда под рукой |
| Кнопка «Позвонить» в мобильной шапке | Параллельный канал для «недоверчивых» |
| Авто-определение языка по геолокации | Если приедут английские туристы — V2 |
| Skeleton-загрузки вместо спиннеров | Воспринимается на 30% быстрее |
| Цены **на каждом блюде**, не «по запросу» | Скрытые цены — топ-1 раздражитель |
| Высота кнопки на mobile — мин. 48px | Зона большого пальца |
| Контрастность текста ≥ 4.5:1 даже на розовых плашках | Реальная читаемость, не «красиво» |
| Изображения с alt и lazy-load | Скорость + SEO + доступность |

## 4. ТЕХНОЛОГИИ

Сравнение трёх стеков для лендинга MOMO. Для каждого — плюсы, минусы, стоимость, скорость запуска, кто это потянет в поддержке.

### 4.1. Вариант A: Next.js + Tailwind + Vercel (рекомендую)

**Что это:** кастомная веб-разработка на современном React-стеке. Vercel — хостинг от создателей Next.js.

| Плюсы | Минусы |
|---|---|
| **Полный контроль** над дизайном — никаких ограничений конструктора | Нужен разработчик для каждой правки |
| **Производительность Lighthouse 95+** — out of the box | Дольше первый запуск (4–6 недель vs 1–2 на Tilda) |
| Можно интегрировать **любой бот, виджет, аналитику** | Хостинг Vercel в России работает с нюансами (домены, СНГ-трафик ок) |
| Хранилище кода в Git → история изменений | Стоимость разработки в 3–5 раз выше Tilda |
| Легко добавлять секции и страницы — V2/V3 не превращаются в боль | |
| SEO-оптимизация автоматическая (SSR, мета, sitemap) | |

**Стоимость (ориентир для лендинга MOMO):**
- **Разработка:** 250 000 — 450 000 ₽ (фрилансер-мидл) или 600 000 — 1 200 000 ₽ (агентство).
- **Vercel:** $0 (Hobby) — $20/мес (Pro). Для одного лендинга и трафика до 100К — бесплатно.
- **Домен .ru:** 200 ₽/год.
- **Поддержка:** 10 000 — 25 000 ₽/мес у разработчика на доработки.

**Когда выбирать:** когда сайт — стратегический актив на годы, есть бюджет на разработку, нужен фирменный визуал и V2/V3 со временем.

### 4.2. Вариант B: Tilda

**Что это:** российский конструктор сайтов, де-факто стандарт для ресторанов и кафе в России. Сосновый Бор и многие другие — на нём.

| Плюсы | Минусы |
|---|---|
| **Запуск за 1–2 недели** | Шаблонный «тильдовый» вид считывается специалистами и частью аудитории |
| Можно править контент **самостоятельно** без разработчика | Сложно сделать **по-настоящему фирменный** визуал |
| Встроенные формы, аналитика, CRM | Lighthouse-оценки 60–80, не 95+ |
| Низкий порог входа | Vendor lock-in: уйти с Tilda = переделать всё |
| Готовые блоки «меню», «контакты», «о нас» | Ограничения по анимациям, кастомным компонентам |
| Интеграции с Яндекс.Картами, ЮMoney, СБП | Если трафик растёт — растёт цена тарифа |

**Стоимость:**
- **Подписка Tilda Personal:** 750 ₽/мес или 5 000 ₽/год (один проект).
- **Tilda Business:** 1 250 ₽/мес или 10 000 ₽/год (несколько проектов).
- **Разработка под ключ:** 60 000 — 150 000 ₽ (фрилансер) или 200 000 — 400 000 ₽ (агентство).
- **Домен:** 200 ₽/год.

**Когда выбирать:** когда нужен быстрый запуск (например, к началу туристического сезона), бюджет ограничен, готовы пожертвовать визуальной уникальностью ради скорости.

### 4.3. Вариант C: Webflow

**Что это:** профессиональный визуальный конструктор, более продвинутый чем Tilda, но требующий обучения.

| Плюсы | Минусы |
|---|---|
| Visual design ближе к кастомной разработке | **Платежи из РФ заблокированы** — нужен VPN, иностранная карта или посредник |
| Можно собрать что-то сильно фирменное без кода | Хостинг Webflow — за рубежом, скорость в РФ переменная |
| Лучше Tilda по гибкости и анимациям | Дороже Tilda (от $18/мес базовый план) |
| Хороший SEO | Меньше готовых блоков «под ресторан», чем у Tilda |

**Стоимость:**
- Hosting Basic: $18/мес (~1 700 ₽ при оплате через посредника).
- Designer: $24/мес (если правите сами).
- Разработка под ключ: 150 000 — 350 000 ₽.

**Когда выбирать:** **не рекомендую для MOMO** из-за платёжных ограничений и переменной скорости из РФ. Webflow упомянут для полноты картины.

### 4.4. Бронирование: интеграции

На MVP-этапе бронирование — **только звонок + форма «Перезвоните мне»** (см. project.md §7). Никаких внешних сервисов на старте.

В V2/V3 рассматриваем апгрейд, если объём звонков превысит возможности администраторов:

## 5. ЦЕННОСТЬ ДЛЯ ГОСТЯ И ЛИДГЕНЕРАЦИЯ

Лендинг ресторана — **не продаёт прямо**. Он создаёт ценность для гостя, формирует доверие и собирает контакты для повторных касаний. «Монетизация» здесь — про **полный жизненный цикл гостя**, а не про прямую продажу.

### 5.1. Воронка ценности

```
ВНИМАНИЕ → ИНТЕРЕС → ПЕРВЫЙ ВИЗИТ → ПОВТОРНЫЙ ВИЗИТ → АДВОКАТ
  ↓           ↓             ↓               ↓                ↓
hero      меню/сезоны    бронь стола    карта лояльности   отзыв/пост
фото      сторителлинг    бот           Telegram-канал     рекомендация
вид моря  детская секция  «маршрут»     email-подписка     UGC
```

На каждом этапе — свой инструмент, своя метрика, своя ценность для гостя.

### 5.2. Инструменты лидгенерации (по приоритету)

#### Приоритет 1: Звонок + форма «Перезвоните мне» (главный лидген)
- **Что даёт ресторану:** прямой контакт администратора с гостем (тёплый канал), номер телефона в форме = база для последующего CRM.
- **Что даёт гостю:** один тап → разговор с живым человеком за 1–2 минуты, мгновенное подтверждение, эмоциональная привязка к месту с первого касания.
- **Метрика:** клики `tel:`, отправленные формы «Перезвоните», % дозвонов, % no-show.

#### Приоритет 2: Telegram-канал «МОМО. Севастополь»
- **Что даёт ресторану:** прямая аудитория, не зависящая от алгоритмов Instagram/VK.
- **Что даёт гостю:** анонсы сезонных меню, событий, джазовых вечеров, мастер-классов.
- **Как продвигать:** ссылка на канал в подтверждении после формы «Перезвоните мне» (письмо/SMS с «спасибо, мы перезвоним. А пока — наш канал»). Плюс QR-код в зале на тейбл-тенте.
- **Контент:** 2–3 публикации в неделю, не больше: смена меню, событие, история блюда.

#### Приоритет 3: Email-подписка на «Сезоны MOMO»
- **Что даёт:** независимый канал, не зависящий от мессенджеров. Конвертирует туристов, которые после поездки хотят «помнить место».
- **Что предлагаем за email:** PDF-гайд *«Гастрономический Севастополь от MOMO»* — 10 локальных продуктов сезона, 3 рецепта, маршрут одного дня. Это **подарок, а не реклама** — отсюда высокая конверсия.
- **Письма:** не чаще 1 в 1.5–2 месяца, под смену сезонного меню MOMO. Если в текущем сезоне MOMO участвует в фестивале «Севастопольские сезоны» — это упоминается в письме как информация, но не как основа канала.

#### Приоритет 4: Сертификаты в подарок
- **Что даёт ресторану:** мгновенная выручка от человека, который, возможно, не придёт сам, но привёл получателя сертификата = чистый прирост клиентов.
- **Цены:** 2 000 ₽ / 5 000 ₽ / 10 000 ₽ / индивидуально.
- **Формат:** электронный (PDF + QR-код) — мгновенно, без визита, с возможностью подписать поздравление.
- **Сезонность:** пик — декабрь (новогодние подарки), март (8 марта), сентябрь–октябрь (дни рождения).

#### Приоритет 5: Программа «именинник» (микро-лидген + UGC)
- Гость указывает дату рождения в форме «Перезвоните мне» (поле опциональное) или администратор спрашивает при первом звонке. За неделю до даты — SMS или email от ресторана: «У нас на дне рождения — бесплатный десерт. Позвоните, забронируем».
- Гость приходит, получает десерт с свечкой и сюрприз-карточкой — **фотографирует и постит**.
- **Что даёт ресторану:** UGC в Instagram/VK + повторный визит + контакт.
- **Что даёт гостю:** ощущение «обо мне помнят».

### 5.3. Партнёрские связки (extra lead-gen)

#### Связка с гастрофестивалем «Севастопольские сезоны»
- MOMO — участник, sevastravel.ru даёт трафик с витрины фестиваля.
- На лендинге MOMO — обратная ссылка и брендирование фестиваля (см. project.md §6.3).
- **Что просить у организаторов:** прямой UTM-link на лендинг, фото-витрина MOMO на главной фестиваля, упоминание в общих рассылках.

#### Связка с отелями Севастополя
- На странице «Маршрут» можно дать партнёрский блок «От нашего ресторана до…» с 4–5 ближайшими отелями.
- В отелях — флаер с QR на лендинг MOMO + промокод *«первый коктейль в подарок»*.
- **Что даёт:** трафик целевой ЦА (туристы), отсюда — бронирования, отсюда — выручка.

#### Связка с экскурсоводами и гидами
- Промо-блок на странице: «Гид? Приходите с группой — для гидов отдельные условия».
- Гиды — мультипликаторы: один гид = 50–200 туристов/сезон.

### 5.4. Метрики ценности (что мерим на сайте)

Простой набор для дашборда:

| Метрика | Зачем | Целевое значение (первые 6 мес) |
|---|---|---|
| Трафик / месяц | База воронки | 5 000+ |
| Конверсия в клик «Бронь» | Главное действие | ≥ 5% |
| % дозвонов + form-completion | Реальный результат | ≥ 90% дозвонов, ≥ 80% форм |
| Клик «Маршрут» | Touristic intent | ≥ 3% |
| Подписки на TG-канал | Долгий лояльный канал | 200+ за 6 мес |
| Подписки на email | Самый автономный канал | 100+ за 6 мес |
| Сертификаты онлайн | Прямая выручка | от 50 000 ₽/мес в сезон |
| Возвраты на сайт | Лояльность | ≥ 25% returning visitors |

### 5.5. Ценность для гостя: «что я получу, открыв этот сайт»

Если перевернуть взгляд с «как ресторан зарабатывает» на «что получает гость», лендинг даёт:

1. **Ощущение места**, ещё не дойдя до него — вид на бухту, интерьер, атмосфера. Это закрывает страх «куплю кота в мешке».
2. **Готовое решение**: «куда повести семью на воскресный обед?» — закрыто за 2 минуты на сайте.
3. **Историю** — почему севастопольская кухня «благородная», что такое «Соль-Сисоль», откуда взялось название MOMO («персик»). Это **повод рассказать друзьям**, причина гордиться выбором ресторана.
4. **Удобство брони** — без звонков, без ожиданий, в чате, который и так открыт.
5. **Память о визите** — TG-канал и email-рассылка дают возможность «забрать кусочек Севастополя» с собой.
6. **Сюрпризы** — детская комната, бесплатный десерт на день рождения, сезонный коктейль — всё это **превышает ожидание** и создаёт момент «вау».

### 5.6. Главные выводы по лидгенерации

1. **Telegram — основной канал**: бот для брони + канал для контента. Бесплатно, контролируемо, прямой контакт с гостем.
2. **Email — резервный канал**, конвертируется через подарок (PDF-гайд), не через «подпишитесь на новости».
3. **Программа лояльности через бота** (V3) — простая stamp-карта или digital-карта. Не делать на старте, дать боту устаканиться.
4. **UGC через персональные сюрпризы** дешевле любой рекламы. Дни рождения, годовщины, «первое свидание» — всё это поводы для постов гостей.
5. **Партнёрства с отелями и гидами** — недооценённый канал. Туристы — 50% потока MOMO, они приходят через отель/гида, не через гугл-поиск.
6. **Не превращать сайт в магазин**. Лендинг = доверие + действие. Магазин = другой контур (как у momokitchen).

## Эксперт 1: UX-дизайнер, 10 лет опыта

### Ошибки, которые делают новички (и которые я вижу в нашей текущей структуре)

1. **Один CTA на весь сайт = «бронь в Telegram».** Это самая частая ошибка ресторанных лендингов: дизайнеры думают «чем проще, тем лучше». Но 30–40% пользователей **не пойдут в Telegram** (нет аккаунта, не доверяют, неудобно с десктопа). На сайте должен быть **второй параллельный канал** — короткая форма «оставьте номер, перезвоним за 15 минут». В RISKS.md это уже зафиксировано (#1.2 и #2.6) — но в project.md пока не дотащено.

2. **Меню как PDF.** В черновике структуры есть кнопка «Открыть полное меню (PDF)». Это **анти-паттерн в 2026** — PDF плохо открывается на mobile, не индексируется поиском, не доступен через скринридеры, обновлять его сложно. Все позиции должны быть на странице как карточки + фильтры.

3. **Карусели.** Если в дизайне появятся карусели отзывов или блюд — это **снижает конверсию** на 15–30% (NN/g данные). Люди не свайпают. Статичная сетка из 3–6 элементов всегда лучше.

4. **Высота шрифта меньше 16px на mobile.** **Первичный сегмент №1 — женщины 45–60**, у многих пресбиопия. Body-text **минимум 17px на mobile**, для меню — 18px. Это не «приятный бонус», это «иначе они не прочитают меню и закроют вкладку».

5. **Контрастность игнорируется ради красоты.** В нашей палитре есть Pink `#E8B6B0` и Teal `#7FB8B0` — на off-white они дают контраст 1.5:1 и 2.0:1 (FAIL по WCAG). В design-research.md §3 это уже зафиксировано, но дизайнеры регулярно нарушают — нужен формальный гейт «контраст-чек перед утверждением макета».

6. **Sticky-элементы, перекрывающие контент.** Sticky-CTA «Бронь» в правом нижнем — ок. Но если поверх ещё cookie-баннер + chat-виджет + WhatsApp-плашка — на mobile экран превращается в кашу. **Правило: максимум один sticky-элемент в зоне большого пальца.**

7. **Hero без обещания.** Хорошо иметь красивое фото бухты, но без явного «семейный ресторан, авторская кухня, вид на бухту, бронь за минуту» — пользователь не понимает, что это вообще.

8. **Сайт «инстаграмного формата» для не-инстаграмной аудитории.** Первичная ЦА — женщины 45–60, которые регулярно ходят сюда годами. Им нужно «знакомое, тёплое, спокойное», а не «вау-новинка». Сайт не должен пытаться выглядеть как «новое модное место» — это противоречит концепции «мы всегда рядом».

### Что ОБЯЗАТЕЛЬНО должно быть

Минимальный набор для семейного ресторана с тремя первичными местными сегментами (плюс к 13 секциям из brief.md §4):

- **Часы работы прямо в hero** или в шапке. Не «о нас → контакты → внизу». Гостья в 12:30 решает «успею пообедать с подругой?» за 3 секунды.
- **Адрес → один тап до карт** (Яндекс / Apple Maps / 2GIS — детект по платформе). См. RISKS.md #2.5.
- **Реальные фото зала и террасы.** Особенно важно — фото с **реальными гостями возраста ЦА**: женщины 45+, не только 25-летние пары. См. `МОМО/интерьер_апрель2026/` — там в кадрах есть гости (на дальнем плане). Нужны кадры с **видимой» аудиторией нашего возраста».
- **Цены прямо в карточках блюд.** Скрытые цены = топ-1 раздражитель.
- **Указание «у нас удобно с детьми»** в hero или сразу после. Молодые родители 25–35 (первичный сегмент №3) проверяют это в первую очередь.
- **Блок «Банкеты» с прямым контактом администратора.** Серьёзная статья выручки — нельзя прятать в подвал.
- **Аллергены на карточках.** Зафиксировано в RISKS.md #2.1 — это законодательная норма + закрытие страха семей.
- **FAQ перед финальным CTA** — 8–9 вопросов, включая про банкеты. Зафиксировано в brief.md.

### Что раздражает и заставляет уйти

1. **Автоплей-видео со звуком.** Запрет — категорический. Если видео-hero (как у Tastavents) — то muted, loop, < 3 МБ.
2. **Попап «оставьте email» через 5 секунд.** На семейном ресторане выглядит как агрессивный спам.
3. **Cookie-баннер на весь экран.** Должна быть тонкая нижняя полоса с «принять/настроить», без блока контента.
4. **Долгая загрузка hero-фото.** На 4G в Севастополе LCP > 3.5s = ~40% bounce. LCP < 2.5s — обязательно.
5. **«У нас лучшие морепродукты в городе!»** Любая non-falsifiable клейма — алерт. Лучше: «Пеленгас от рыбаков с Балаклавы, привозим утром».
6. **Несвежесть в digital-следах.** Если в Instagram пост последний был 4 месяца назад, а на сайте «свежие новости», доверие падает мгновенно. **Сайт = живой ресторан**. Если не готовы поддерживать TG-канал и Instagram — лучше эти блоки убрать с сайта совсем.

## Эксперт 2: Маркетолог, специалист по конверсии (CRO)

### Какой оффер зацепит

Семейный ресторан — это **четыре разных оффера для четырёх сегментов**, не один. Сегменты обновлены 2026-05-20 по концепции владельца: первичная аудитория — местные, туристы вторичный канал.

| Приоритет | Сегмент | Оффер | CTA |
|---|---|---|---|
| **Первичный 1** | **Женщины 45–60** (педагог, бухгалтер, риелтор) | «Тёплое место, в которое возвращаются. Спокойный обед или ужин с подругами» | Coral CTA «Встретиться с подругами» |
| **Первичный 2** | **Молодые пары 20–27** | «Стол у окна, мягкий свет, тихий джаз. Без лишнего внимания — только вы и она» | Coral CTA «Романтический вечер» |
| **Первичный 3** | **Молодые родители 25–35** | «Детская комната, детское меню. Друзья за столом, дети — за раскрасками» | Coral CTA «Семейный обед» |
| Вторичный | **Турист с семьёй** | «Закат над Севастопольской бухтой. Стол у окна, рейтинг 4.9 на Яндексе» | Coral CTA «Стол у окна» |

**Реализация:** один Hero, но **4 карточки-входа «Сегодня я…»** под ним: «Встречаюсь с подругами» / «На свидание» / «С детьми или с семьёй» / «На сезонный сет». Каждая ведёт к секции с релевантным оффером. Это **сегментирующий лендинг** — конверсия на 15–25% выше «универсального» (Unbounce, CXL данные).

**Отдельный канал — банкеты.** MOMO активно проводит свадьбы, юбилеи, семейные торжества и поминки. Это **серьёзная статья выручки**, не побочный канал. На главной странице — якорный блок «Банкеты и торжества» (см. brief.md §5.9.1) с прямым контактом администратора. Лид-магнит: «Расскажите формат — подберём предложение за день» (низкий barrier, конкретное обещание).

**Замечание по UX-сценариям §3** (Алексей, Мария, Игорь): персонажи покрывают сегменты «турист с семьёй», «молодая пара» и «молодые родители». **Не покрыт первичный сегмент «Женщины 45–60»** — добавить отдельный сценарий «Светлана, 52, встреча с двумя подругами в среду в 13:00» в V2 при доработке UX-карты.

**Главный CRO-вывод по новой сегментации.** Раньше Hero оптимизировался под туриста («вид на бухту»), теперь — под три параллельных оффера. Hero **должен быть нейтральным** (не «для туриста», не «для семьи», не «для свидания»), а сегментирующий блок «Сегодня я…» — **главный CRO-инструмент** этого лендинга. Конверсия на лендингах с сегментацией под несколько ЦА на 15–25% выше, чем «универсальный оффер» (Unbounce, CXL).

### Триггеры доверия

1. **Социальное доказательство в hero.** «★ 4.9 на Яндекс.Картах, 2045 отзывов, с 2013 года» — три цифры в одной строке. Не «★ 5.0» — слишком идеально, читается как фейк (RISKS.md #4.5).
2. **Логотипы публикаций / партнёров.** «Участник гастрофестиваля Севастопольские сезоны 2026», «Партнёр sevastravel.ru». Авторитет переносится.
3. **Реальные имена и фото.** Шеф-повар с именем и историей — не «наша команда».
4. **Гарантия отмены.** «Бесплатная отмена до 4 часов до визита». Снимает страх «а если что».
5. **Транспарентность цены.** «Средний чек 1500 ₽/гость» прямо в hero. Скрытые цены = недоверие.
6. **Микрокопия под CTA.** «Ответим за 5 минут. Без предоплаты». Каждая фраза снимает одно конкретное сомнение.
7. **Полная контактная информация.** Адрес, телефон, юр.лицо — это всё ещё работает как trust-сигнал, особенно для местных.

### Путь посетителя «зашёл → оставил заявку»

Восемь шагов с **точкой обрыва на каждом** и метрикой:

```
1. ЗАШЁЛ ─────────────► Hero за 3 сек: «понял, куда попал»
     ▼ обрыв (30% уходят)            метрика: bounce < 50%
2. ПРОЛИСТАЛ Hero ────► карточки «Сегодня я...»
     ▼ обрыв (15%)                    метрика: scroll-depth > 40%
3. ВЫБРАЛ СЕБЯ ───────► попал в релевантный блок:
                          • меню / о нас (подруги, свидание)
                          • детское меню (семья с детьми)
                          • сезонный сет
                          • банкеты (мини-банкет, торжество)
     ▼ обрыв (10%)                    метрика: клик по карточке > 35%
4. ПОНЯЛ ЦЕНУ/ПРЕДЛОЖ. ► увидел средний чек, фото, отзывы
     ▼ обрыв (20%)                    метрика: время на блоке > 30 сек
5. ПОСМОТРЕЛ FAQ ─────► снял возражения (дети, парковка, отмена, банкеты)
     ▼ обрыв (5%)                     метрика: FAQ engagement > 25%
6. КЛИКНУЛ CTA «БРОНЬ» ► tel: или форма «Перезвоните»
     ▼ обрыв (15%)                    метрика: CTR на CTA > 8%
7а. ПОЗВОНИЛ ────────► администратор поднял трубку
     ▼ обрыв (10% — не дозвон)        метрика: % дозвонов > 90%
7б. ОТПРАВИЛ ФОРМУ ──► заявка ушла администратору
     ▼ обрыв (5%)                     метрика: form-completion > 80%
8. ПОЛУЧИЛ ПОДТВ. ───► бронь зафиксирована
```

**Отдельная воронка для банкетов** (длинный цикл, не закрывается в одном сеансе):
```
1. ЗАШЁЛ ──► 2. КЛИКНУЛ КАРТОЧКУ «Семейный обед / банкет»
   ──► 3. ПРОЧИТАЛ БЛОК «Банкеты» ──► 4. ПОЗВОНИЛ
   ──► 5. РАЗГОВОР С АДМИНИСТРАТОРОМ (5–15 мин)
   ──► 6. ВСТРЕЧА / ПРЕДОПЛАТА (вне сайта)
   ──► 7. ТОРЖЕСТВО ПРОВЕДЕНО
```
Тут метрики: **% кликов по карточке-полоске «Банкеты», % дозвонов от тех, кто прочитал блок**, конверсия из звонка в банкет (отслеживается администратором офлайн).

**Главные точки обрыва — 1, 4, 6, 7а.** Что чинить в первую очередь:
- **Точка 1 (hero):** добавить адрес + часы + рейтинг + 3-секундное обещание.
- **Точка 4 (цена):** показать «от 1500 ₽» явно. Сейчас в project.md этого нет.
- **Точка 6 (CTA):** sticky + 5 повторов по странице (design-research.md §8).
- **Точка 7а (звонок):** обеспечить, чтобы администратор поднимал трубку. Метрика «% дозвонов» — ключевая. Если падает ниже 85% — нужен дополнительный администратор или Restoplace.

### Что добавит конверсию на 15–30%

1. **Точная микрокопия CTA.** Не «Забронировать», а «Забронировать · +7 978 855-75-54 · 1–2 гудка». Номер прямо в кнопке.
2. **Лид-магнит для email-сегмента.** PDF «Гастрономический Севастополь от MOMO» в обмен на email (research.md §5.2). Конвертирует туристов после поездки.
3. **Кнопка «Подарочный сертификат»** в шапке — отдельная воронка, низкий barrier (RISKS.md #3.10).
4. **Re-engagement через TG-канал.** Если пользователь пришёл, не забронировал, но подписался на канал — это **отложенная конверсия** через 1–3 месяца.

## Эксперт 3: Светлана, 52 года — постоянная гостья MOMO (первичная аудитория №1)

> Эксперт 3 переписан 2026-05-20. Прежняя персона «владелец бизнеса в командировке» (35–55) была сдвинута во вторичную аудиторию (туристы). На её место — главный сегмент MOMO: **женщина 45–60, местная, постоянная гостья**. Многие выводы прежнего эксперта валидны и здесь (профессиональные фото, без пафоса, не Tilda 2018) — они сохранены, но переформулированы через лицо Светланы.

**Кто такая Светлана.** 52 года, главный бухгалтер строительной компании, высшее образование, живёт в Севастополе с детства. Ходит в MOMO **с 2014 года**, минимум раз в две недели — обедает с подругами, отмечает дни рождения коллег, иногда приходит с мужем поужинать в субботу. Знает многих официантов по имени. Дочь 24 года иногда зовёт её сюда на свидание — «мам, ты же давно не была».

Светлана **не зайдёт на сайт MOMO просто так**. Зайдёт в трёх ситуациях:
- посмотреть актуальные часы перед встречей с подругой;
- скинуть подруге ссылку на меню;
- забронировать стол на 6 человек для дня рождения коллеги или поминок по сестре.

### Первые 5 секунд: на что смотрит

Это не «исследование», это **проверка**: «не сломалось ли что у моего любимого места».

| Что видит | Что думает |
|---|---|
| Фото зала (хорошо ли знакомое) | «Те же диваны, тот же закат — да, оно» |
| Часы работы | «До скольких сегодня?» |
| Телефон | «Это тот же номер, или поменяли?» |
| Меню — узнаваемые названия | «Где Соль-Сисоль? Есть. Хорошо» |
| Кнопка «Банкеты» | «А, можно у них на 12 человек заказать?» |

**Если что-то выглядит «не MOMO» — Светлана пугается.** Слишком модный hero, незнакомый шрифт, странные пастельные цвета — она подумает, что сменился владелец или концепция, и **позвонит уточнить, не закрылись ли**. Сайт должен **подтверждать узнавание**, не «обновлять впечатление».

### Что заставит остаться (плюсы)

1. **Узнаваемые фото зала** — те же диваны, тот же закат. У нас они есть (`МОМО/интерьер_апрель2026/`).
2. **Телефон большой и видный** — Светлана звонит, не набирает форму.
3. **Понятное меню с ценами и граммовкой** — она привыкла к этому формату, лишние изменения пугают.
4. **Блок «Банкеты» с прямым контактом администратора** — это сценарий, который **сайт раньше не закрывал**, а Светлана его регулярно использует.
5. **Реальные имена** в команде. «Знакомая официантка на фото» = доверие выше.
6. **Простая структура** — Светлана **не ищет** «новинки», она ищет «то же, что в прошлый раз».

### Что заставит закрыть вкладку

1. **«Перезагрузка концепции» в визуале** — слишком модный hero с инстаграм-эстетикой, незнакомые шрифты, агрессивные акценты. Воспринимает как «всё поменялось — не моё больше». Решение: тёплая дневная атмосфера в Hero (IMG_8853 хорош), серьёзный антиквенный сериф, не «trendy».
2. **Сайт «для туристов»** — если на главной только закат и «авторская кухня для гостей города», Светлана чувствует «теперь они забыли про нас». Решение: сегментирующий блок «Сегодня я…» с первой карточкой **«Встречаюсь с подругами»**, а не «На вечер с видом».
3. **Любительские фото блюд** — она много раз видела эти блюда вживую, плохая фотография = «у них теперь и кухня хуже?». Решение: фото блюд из `Меню кухня/` достаточно качественные.
4. **Узкие часы дозвона.** «Позвоните с 10 до 18» — Светлана работает до 18. Решено формой «Перезвоните мне» с окном «когда удобно».
5. **«Лучший ресторан Севастополя»** или подобный пафос. За 10 лет дружбы с MOMO она знает их сильные и слабые стороны — пафос звучит как «нас подменили».
6. **Тильда-блоки 2018 года.** Светлана видит много сайтов — узнаваемая Tilda-эстетика читается как «сэкономили, не вложились».
7. **Никаких отзывов** или одна звезда 5.0. Слишком идеально = подозрительно (RISKS.md #4.5). 4.9 с 2045 отзывов — то, что надо.

### Что НЕ важно Светлане

- Английская версия (она читает по-русски).
- Доставка (она ходит в зал, не заказывает).
- Сезонный сет фестиваля (она читает основное меню — фестиваль ей «модность»).
- Программа лояльности через приложение (карточку в кошельке — да, скачивать приложение — нет).
- Соцсети — на VK она подписана, но новые лендинговые посты её не интересуют.

### Что Светлана скажет подруге после того, как зашла на сайт

«Глянь, у нас MOMO сайт сделал — вроде ничего, фотки красивые. На сезонный сет надо как-нибудь сходить, я их раньше не пробовала. Ой, у них и банкеты можно заказать — ты помнишь, твоей маме скоро 70?». Это и есть **главный успех сайта для этого сегмента** — не «вау, как красиво», а **«можно поделиться без стыда» + узнавание новых возможностей** (банкеты, сезон) у знакомого места.

### Дополнительная персона — деловой профессионал 35–55 (вторичная аудитория)

Прежняя персона эксперта 3 сохраняет валидность для **вторичного сегмента** «туристы с покупательской способностью» / «командировочные»:
- Профессиональные фото, конкретная локация и парковка, карта вин с ценами — всё это **тоже работает на Светлану**, поэтому ничего не теряется.
- Английская версия и «подходит для встречи с партнёром» — закрывает **только этот сегмент**, не Светлану. Поэтому английский — fast-follow MVP, не V2 (он реально нужен 10–20% трафика).
- Принципы «без пафоса, без Tilda 2018, без 5.0 рейтинга» — универсальны для обоих сегментов.

### Сделать в первые 1–3 месяца после запуска

14. **TG-канал MOMO** с регулярным контентом — иначе сайт «врёт о живости» (Эксперт 1, пункт 6).
15. **Подарочные сертификаты** на сайте — отдельная воронка (хорошо ложится на сегмент «женщины 45–60» — подарок коллегам).
16. **PDF «Гастрономический Севастополь»** в обмен на email — конвертирует туристов (вторичный сегмент).
17. **3D / 360°-превью террасы** — догоняет Мекензи.
18. **Бесплатная отмена до 4 часов** как явная политика.

### Принципы, которые нельзя нарушать

- Не делать **один универсальный оффер** — он размывает все три первичных сегмента.
- Не делать **Hero «для туристов»** (с эксклюзив-обещаниями) — это отчуждает Светлану и её сегмент.
- Не использовать **Pink и Teal как текст** — только как заливки плашек (design-research.md §3).
- Не **прятать цены** ни в одном виде.
- Не **прятать часы, адрес и телефон** глубже первого экрана.
- Не **обещать** «лучшее в Севастополе» / «уникальный опыт» — это **анти-доверие** для всех трёх экспертов.
- Не **прятать блок «Банкеты»** — это серьёзный канал выручки.
- Не **запускать без формы «Перезвоните мне»** — теряем людей вне рабочих часов.
- Не **«обновлять впечатление» через модный визуал** — постоянные гости пугаются.

## 1. Пять кейсов Telegram Mini App в ресторанной нише

### 1.1. Dodo Pizza — TMA трекинга и заказа
**Тип:** Сеть пиццерий, ~1000+ точек.
**Что делает:** Заказ пиццы, выбор адреса доставки, реал-тайм трекинг курьера, история заказов, программа лояльности.
**Сильные стороны:** Заточен под повторный заказ — за 2 тапа можно повторить прошлый. Push от бота о статусе («собираем», «в пути»). Карта с курьером на финальном этапе. Brand-tone выдержан — pizza-tracker как «секундомер удовольствия».
**Слабые стороны:** На старте TMA было «легко поскользнуться» — длинные категории меню в виде гигантской прокручиваемой простыни, без чёткой фиксации шапки. Цены менялись при выборе адреса (логика верная, но без явного сообщения — пользователи паниковали).
**Результаты по публикациям:** −20% среднее время ожидания, +30% повторных заказов в первые 3 месяца после внедрения.
**Что МОМО может взять:** Идея «повтор прошлого заказа в 2 тапа». Под МОМО это переформулируется в **«повторить прошлую бронь»** — кнопка «Как в прошлый раз: стол у окна, 19:30, на двоих, годовщина». Огромный win для постоянных гостей (особенно сегмента «Светлана 52»).
**Что НЕ брать:** Маршрут курьера — у МОМО нет доставки. Длинные простыни меню тоже не повторять.

### 1.2. Burger Brothers — TMA автоматизации заказов
**Тип:** Сеть бургерных в РФ.
**Что делает:** Меню, заказ, оплата, push-уведомления о статусе, рассылка о спецпредложениях.
**Сильные стороны:** Полная автоматизация — заказ принимается ботом, не оператором; экономия на людях. Прямой канал к лояльной аудитории через рассылку (без алгоритмов Instagram/VK).
**Слабые стороны:** Минималистичный UI на грани «дешёво» — карточки без хорошего фото, простой шрифт. Видно, что цель — функциональность, не атмосфера.
**Результаты по публикациям:** Доля прямых заказов выросла с 48% до 70% (отъели у Delivery Club / Яндекс.Еды), за счёт чего экономия на комиссии агрегаторам ~20–30%.
**Что МОМО может взять:** **Прямой канал коммуникации без агрегаторов** — для МОМО это перевод существующих гостей с «позвонить» на «открыть Mini App». Главное — не переплачивать за доставку-инфраструктуру, которая нам не нужна.
**Что НЕ брать:** Минимализм-«дёшево» — для МОМО критично сохранить атмосферу.

### 1.3. Шаверма от Шера — TMA для быстрого заказа
**Тип:** Сеть шавермы.
**Что делает:** Меню, заказ навынос, оплата, текущие акции на главной.
**Сильные стороны:** Очень короткий путь до оформления — 2–3 экрана. Идеально для нишы «голодный, нет времени». Промо-баннер на главной мотивирует ситуативный заказ.
**Результаты по публикациям:** +22% заказов в первый месяц, +15% средний чек (за счёт upsell в меню «возьмите ещё»).
**Что МОМО может взять:** **Промо-баннер на Home** «Сезонное меню Севастопольские сезоны до 31 мая». Короткий путь от Home до целевого действия (max 2 тапа до брони).
**Что НЕ брать:** Тон «фастфуд» — для МОМО неприемлем.

### 1.4. Restoplace TMA — виджет бронирования
**URL setup:** [restoplace.cc/help/tg-mini-app](https://restoplace.cc/help/tg-mini-app)
**Тип:** SaaS-виджет, встраивается в TMA любого ресторана.
**Что делает:** Бронь столов / банкетов, продажа сертификатов и билетов, депозиты под бронь.
**Сильные стороны:** Готовое решение «под ключ» — ресторан не пишет код, только конфигурирует в BotFather. Один и тот же виджет работает в web и TMA. Бронь с депозитом снижает no-show.
**Слабые стороны:** Брендирование ограничено — выглядит «как у всех клиентов Restoplace». Кастомные UX-фишки невозможны. Зависимость от поставщика (vendor lock-in).
**Что МОМО может взять:** **Параллельная установка Restoplace** как fallback-канала брони к телефону. На MVP это даёт возможность брони онлайн, не разрабатывая свой бот.
**Что НЕ брать:** Полная зависимость от их UI — не даёт фирменный стиль МОМО.

### 1.5. tgEDA / Smartofood / Sandyapp / Q3 Service — экосистемные платформы
**URLs:** [smartofood.ru/miniapp](https://smartofood.ru/miniapp/), [sandyapp.ru/telegram](https://sandyapp.ru/telegram), [tg-order.ru](https://tg-order.ru/article-telegram-restaurant-app.html), [q3-service.ru](https://q3-service.ru/)
**Тип:** B2B-платформы «TMA для ресторанов под ключ».
**Что делают:** Готовый TMA с интеграцией iiko/Frontpad/r_keeper. Меню, заказ, оплата, бронь, лояльность, рассылки.
**Сильные стороны:** Не нужен разработчик — ресторан загружает меню и работает. Интеграция с POS-системами (важно для сетей).
**Слабые стороны:** Шаблонный вид, узнаваемая «tgEDA-эстетика» — гости видят сотни таких приложений и устают. Подписка ежемесячно (4–10 К ₽/мес). Сложно сделать что-то фирменное.
**Результаты по публикациям:** tgEDA на август 2024 — ~800 ресторанов, 150–800 заказов/мес/точка после первого месяца. Один из клиентов (без названия по NDA) — 11 000 MAU, 200+ функций.
**Что МОМО может взять:** **Сравнить с своим решением по экономике**. Если разработка кастомного TMA ~150–300 К ₽ единоразово, а tgEDA ~6 К ₽/мес = 72 К ₽/год — окупаемость кастома за 2–3 года. Кастом стоит, только если фирстиль важен (а для МОМО — да).
**Что НЕ брать:** Готовое решение «как у всех» — это понижает воспринимаемое качество, что критично для сегмента «Светлана 52».

### Сводка по кейсам

| Кейс | Главный win | Что взять для МОМО |
|---|---|---|
| Dodo Pizza | Повтор прошлого заказа в 2 тапа | Кнопка «Повторить мою прошлую бронь» |
| Burger Brothers | Прямой канал, нулевая комиссия | TMA как замена пути «позвонить» для местных |
| Шаверма от Шера | Промо-баннер ситуативно | Баннер сезонного меню на Home |
| Restoplace | Готовая бронь с депозитом | На MVP подключить как fallback |
| tgEDA / Smartofood | Дешёвый старт без разработки | Сравнить экономику, понять — кастом или нет |

**Главный вывод по кейсам:** ни один из публично разобранных не закрывает сценарий **«семейный ресторан, основная аудитория — постоянные гостьи 45–60, цель TMA — НЕ доставка/заказ, а удобство брони + сезонное меню + банкеты»**. МОМО в этой нише может стать первым, кто делает TMA-витрину «премиум-семейного ресторана», а не «доставочную точку».

## 2. Обязательные экраны и UX-паттерны TMA для ресторана

### 2.1. Типовая структура (8–10 экранов)

Учитывая UX-специфику TMA (полноэкранный модал, BackButton в шапке Telegram, MainButton снизу, нет «scroll to top», haptic, theme adaptation), оптимальный набор — **9–11 экранов** с одноуровневой навигацией от Home.

1. **Home / Hero** — логотип, краткий слоган, 1 главное фото + 3–4 крупные плитки-сегменты («Сегодня я… отдыхаю с семьёй / встречаюсь с друзьями / устраиваю праздник»). **MainButton:** «Забронировать стол». Должно умещаться в 1–1.5 экрана: пользователь не любит долгий скролл в модале.
2. **Меню — Кухня (категории)** — список 8 подкатегорий (Закуски / Салаты / Супы / Пасты / Мясо / Рыба / Римская пицца / Десерты), каждая — горизонтальная карточка с иконкой/фото + счётчик блюд.
3. **Подкатегория меню** — сетка 1 или 2 колонки: фото, название, описание (1 строка), цена, вес. Тап по блюду → **bottom sheet** с большим фото, полным описанием, аллергенами. Чипсы-фильтры сверху: «Острое», «Веган», «Детям ок».
4. **Меню — Бар** — отдельно от кухни: коктейли, согревающие, вино. Та же логика, визуально темнее/премиальнее.
5. **Детское меню** — отдельный экран, лёгкая графика, большие фото + блок «Что у нас для детей» (стульчики, раскраски, игровая зона).
6. **Сезонное меню («Севастопольские сезоны»)** — промо-экран с акцентом на текущий сезон. Большое фото, 4–8 блюд, описание концепции. Эмоциональный экран — фото большие, текста мало.
7. **О нас / Концепция** — короткий текст (2–3 абзаца), фото команды/интерьера, блок «Наши ценности» (3–4 пункта с иконками).
8. **Галерея** — сетка фото 2 колонки, тап → полноэкранный просмотр со свайпом (через `WebApp.expand()`). Опционально табы «Зал / Терраса / Атмосфера».
9. **Бронирование** — главный конверсионный экран. Кнопка «Позвонить» + короткая форма «Перезвоните мне» (имя + телефон + время — 3 поля макс). **MainButton:** «Отправить заявку» → `HapticFeedback.notificationOccurred('success')` + `WebApp.close()`.
10. **Контакты и как добраться** — адрес, телефон (тап → звонок), часы, кнопка «Открыть в Яндекс.Картах» через `WebApp.openLink()`. Статичная картинка карты (iframe в TMA глючит). Соцсети через `WebApp.openTelegramLink` / `openLink`.
11. **Банкеты** (опционально, можно мерджить в «Бронь») — описание банкетных возможностей, фото, CTA «Обсудить банкет» через `openTelegramLink` к менеджеру.
12. **FAQ** (опционально) — аккордеон 6–10 вопросов. Если коротко — можно убрать в «О нас» / «Контакты», экономя экран.

**Что НЕ делать отдельным экраном:** футер, cookie-policy, copyright — это веб-артефакты. В TMA достаточно мелкого текста на «О нас».

### 2.2. Must-have элементы (без них не работает)

- **Телефон по тапу** (`tel:`) — основной способ брони для МОМО.
- **Адрес + кнопка построить маршрут** (deep link в карты через `WebApp.openLink`).
- **Часы работы** — желательно с автоопределением «Открыто сейчас / закрыто».
- **Меню с фото и ценами** — главная причина, по которой открывают.
- **`BackButton`** настроен на каждом не-Home экране через `WebApp.BackButton.show()` + `onClick`.
- **`MainButton`** на ключевых экранах (Home → «Забронировать», Бронь → «Отправить»).
- **Theme adaptation** — использовать `WebApp.themeParams` через CSS-переменные. В тёмном режиме Telegram сайт должен быть тёмным.
- **`WebApp.ready()`** + **`WebApp.expand()`** сразу — иначе app откроется в половину экрана.
- **Корректная работа без интернета на старте** — skeleton, не белый экран.

### 2.3. Nice-to-have (+20% удобства)

- `HapticFeedback` на тапах по кнопкам.
- Фильтры в меню (веган / без глютена / детское).
- Избранные блюда (`CloudStorage`, без бэкенда).
- Поиск по меню (если позиций > 60).
- Промо-баннер на Home (сезонное меню).
- Кнопка «Поделиться рестораном» через `WebApp.switchInlineQuery` — органический рост.
- `CloudStorage` для брони / истории / избранного между устройствами.
- `SettingsButton` в шапке для «Связаться / О приложении».

### 2.4. Антипаттерны (НЕ делать)

- Длинные формы (макс 3 поля).
- Свои модалки/попапы поверх контента — Telegram уже модал. Использовать `WebApp.showPopup()` / `showAlert()`.
- Cookie-banner / GDPR — в TMA не нужен.
- Sticky-хедер с навигацией — дублирует Telegram-шапку, съедает 60–80 px.
- Кнопка «Наверх» — занимает место, сбивает с MainButton.
- Свой Back-button в UI — есть `WebApp.BackButton`.
- iframe-карта Яндекса — глючит в webview, использовать статичную картинку.
- Авто-открытие модалок при загрузке (подписка, акция) — повышает закрытие.
- Тяжёлые анимации, parallax, scroll-jacking — лагают на средних Android.
- Внешние ссылки без `WebApp.openLink()` — `target="_blank"` открывается криво.
- Регистрация / login-формы — есть `WebApp.initDataUnsafe.user`.
- Inline-видео с автоплеем со звуком — мгновенный close.

## 3. Технические возможности TMA: чего НЕТ у обычного сайта

Все доступно через `window.Telegram.WebApp` или обёртки SDK. Версии API указаны для `WebApp.isVersionAtLeast()` — старые клиенты молча игнорируют новые методы.

| # | API | Что даёт | Версия | Полезно МОМО? |
|---|---|---|---|---|
| 1 | **`initData` / `initDataUnsafe`** | Подписанный HMAC пакет с `user.id`, `first_name`, `username`. Сервер валидирует — **бесплатная авторизация без OTP** | 6.0 | **Да** — база всего |
| 2 | **`BackButton`** | Системная кнопка «назад» в шапке Telegram | 6.1 | Да |
| 3 | **`MainButton` / `SecondaryButton`** | Закреплённая снизу кнопка с прогрессом и haptic | 6.1 / 8.0 | **Да** |
| 4 | **`SettingsButton`** | Шестерёнка в меню «...» | 6.1 | Опционально |
| 5 | **`HapticFeedback`** | Тактильные отклики на тапы | 6.1 | **Да** |
| 6 | **`ThemeParams` + `colorScheme`** | Цвета и тема Telegram (light/dark) — авто-адаптация | 6.0 | **Да** |
| 7 | **`CloudStorage`** | Key-value хранилище по user_id. До 1024 ключей × 4096 байт. **Без своего backend** | 6.9 | **Да** — лояльность без БД |
| 8 | **`requestContact()`** / **`requestWriteAccess()`** | Получить телефон + право писать пользователю | 7.2 | **Да** — бронь без формы |
| 9 | **`openInvoice(url)`** | Нативный платёжный шит Telegram (Bot Payments / Stars) | 6.1 | **Да** — депозит банкета, сертификаты |
| 10 | **`shareToStory(media)`** | Создаёт Telegram Story с медиа и кнопкой обратно | 7.8 | **Да** — органический трафик |
| 11 | **`switchInlineQuery(query)`** | Перебросить в чат с готовым inline-запросом | 6.7 | Полезно V2 |

## 5. Стек, фреймворки, инфраструктура, платежи

### 5.1. NPM-пакеты / фреймворки

| Пакет | Что это | Когда брать |
|---|---|---|
| **[`@telegram-apps/sdk`](https://github.com/Telegram-Mini-Apps/telegram-apps)** + **`@telegram-apps/sdk-react`** | Современный комьюнити-SDK, TypeScript-first, реактивные сигналы, отдельные пакеты для React/Vue/Solid | **Дефолтный выбор в 2026** |
| **[`@twa-dev/sdk`](https://github.com/twa-dev/SDK)** | Тонкая TS-обёртка над `window.Telegram.WebApp`, типы + удобный импорт | Если нужен минимум абстракций |
| **[`@telegram-apps/telegram-ui`](https://github.com/Telegram-Mini-Apps/TelegramUI)** | UI-кит в стиле Telegram (iOS/Material), auto-тема | Не для МОМО — у нас свой брендинг |
| **`@telegram-apps/init-data-node`** | Серверная валидация `initData` (HMAC) | Всегда, если есть backend |
| **[grammY](https://grammy.dev)** (Node TS) или **[aiogram 3.x](https://docs.aiogram.dev)** (Python) | Фреймворк для бота | grammY если стек Node; aiogram если Python |

Стартер: `npm create @telegram-apps/mini-app@latest` — официальный CLI, поднимает Vite + React + sdk-react + telegram-ui за 30 секунд.

### 5.2. Инфраструктура

**MVP без backend (для МОМО реально):**
1. **Бот через [@BotFather](https://t.me/BotFather)**: `/newbot` → `BOT_TOKEN`. Затем `/newapp` → привязать URL TMA. Меню-кнопка: `/setmenubutton` → ввести URL и подпись («Открыть МОМО»).
2. **Хостинг TMA**: Vercel / Cloudflare Pages / Netlify — любой статичный HTTPS-хостинг. **У МОМО уже Vercel** — можно тот же проект развернуть на сабпасе `/app`.
3. **Deep links**: `https://t.me/momo_sev_bot/app?startapp=banquet_12` — глубинные ссылки в конкретный экран TMA через `WebApp.initDataUnsafe.start_param`. Полезно для QR-кодов в зале.

**Когда нужен backend (V2):**
- Валидация `initData` (HMAC) на каждом запросе.
- Хранение броней, отзывов, истории (CloudStorage хватит до ~1000 пользователей с минимальным стейтом).
- Webhook от Bot Payments (`pre_checkout_query`, `successful_payment`).
- Стек: grammY на Cloudflare Workers + D1 (SQLite) бесплатно тянет нагрузку МОМО.

### 5.3. Платежи в РФ — главный больной вопрос 2026

- **[ЮKassa](https://yookassa.ru)** (бывшая Яндекс.Касса) — исторически основной провайдер Bot Payments в РФ. Проверять актуальный статус в BotFather перед стартом.
- **Сбербанк, Тинькофф (Т-Банк)** — напрямую через Bot Payments официально не интегрированы; через сторонних агрегаторов.
- **Telegram Stars** (валюта `XTR`) — работает в РФ, но **по правилам Telegram только для цифровых товаров**. Для сертификатов МОМО — серая зона по 54-ФЗ (онлайн-касса не пробьёт чек).
- **Прагматичная схема:** для оплат с фискализацией — собственная интеграция с ЮKassa/CloudPayments напрямую (без Telegram-шита), открывать `WebApp.openLink(invoice_url)`. Потеря нативного UX, но соблюдение 54-ФЗ.

**Перед V2 рекомендую** позвонить в поддержку ЮKassa: «работает ли Bot Payments для юрлица в РФ + фискализация по 54-ФЗ». Эта часть меняется быстро.

### 5.4. Аналитика

- **`@telegram-apps/analytics`** (официальный) — встраивается за 5 минут.
- **TGStat** — внешняя аналитика по боту.
- **Своя**: эндпоинт + Plausible / Umami по `initDataUnsafe.user.id` (без cookies).

Стандартные Google Analytics / Яндекс.Метрика часто блокируются в webview и тормозят — лучше не использовать.

<a id="tma-6-adapt"></a>
## 6. Адаптация существующего веб-лендинга МОМО в TMA

**Короткий ответ:** технически можно «как есть» — TMA это webview, любой URL грузится. **Но** «как есть» = плохой UX. Нужны 10–15 точечных адаптаций, не полная переделка.

### Подключение Telegram SDK

```html
<script src="https://telegram.org/js/telegram-web-app.js"></script>
```

На старте:
```js
Telegram.WebApp.ready();
Telegram.WebApp.expand();
Telegram.WebApp.disableVerticalSwipes(); // чтобы свайп вниз не закрывал
```

### CSS theming (критично)

Завести CSS-переменные `--tg-bg`, `--tg-text`, `--tg-button`, `--tg-hint`, `--tg-link` и маппить их из `Telegram.WebApp.themeParams` при загрузке + `themeChanged`. Палитра МОМО (coral/teal/navy/pink/oak) остаётся как акценты, но **базовые фон/текст должны следовать Telegram-теме** (свет/тёмная).

### Навигация

- **Убрать sticky-header** — дублирует Telegram-шапку, забирает 60–80 px.
- Логотип оставить, но не sticky — в начале Home.
- Якорные ссылки → переключить на router (hash-based), чтобы BackButton работал.
- На под-экранах: `BackButton.show()` + `onClick(() => history.back())`. На Home: `BackButton.hide()`.

### CTA / кнопки

- Все CTA «Забронировать» → переключить на `Telegram.WebApp.MainButton`:
  ```js
  MainButton.setText('Забронировать стол');
  MainButton.show();
  MainButton.onClick(() => router.go('/booking'));
  ```
- `tel:` ссылки работают как есть.
- Внешние ссылки → обернуть в `Telegram.WebApp.openLink(url)` или `openTelegramLink` для `t.me`.

### Что убрать в TMA-версии

- Cookie-banner — не нужен.
- Кнопка «Наверх» / scroll-to-top — лишняя.
- Дублирующая шапка-навигация — заменить на BackButton.
- Тяжёлый footer — оставить мини-блок «© МОМО 2026», остальное скрыть.
- Все popup/модалки своей реализации → заменить на `WebApp.showPopup` / `showAlert`.

### Формы

- «Перезвоните мне» — оставить, но 2–3 поля максимум.
- После submit: `HapticFeedback.notificationOccurred('success')` + `WebApp.showAlert('Спасибо, перезвоним в течение 15 минут')` + `WebApp.close()`.
- Использовать `Telegram.WebApp.initDataUnsafe.user` для предзаполнения имени/телефона.

### Производительность

- HEIC/JPG галереи → WebP (уже частично есть jpeg).
- Lazy-loading через `loading="lazy"`.
- Inline критический CSS, отложить остальное.
- Шрифты — `font-display: swap`. System-fonts вместо Google Fonts — webview грузит медленнее.

### Карта

- Заменить embed-iframe Яндекс.Карт на статичное изображение + кнопка «Открыть в Яндекс.Картах» через `openLink`.

### Viewport и safe-area

```html
<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">
```

```css
padding-bottom: calc(env(safe-area-inset-bottom) + 84px);
/* 84px — высота MainButton */
```

### Объём работы

- Сам HTML/CSS лендинга остаётся **на 80–90%** (контент, стили, картинки).
- Добавляется **тонкий слой адаптации** (~200–400 строк JS): инициализация WebApp, theme-маппинг, BackButton/MainButton hookup, замена popup'ов, прокидывание ссылок через `openLink`.
- **Итог:** не «всё переделать», а «обернуть в TMA-shell + 12 точечных правок».

<a id="tma-7-verdict"></a>
## 7. Финальный вывод для МОМО

### Ради чего вообще делать TMA для МОМО (4 главных win)

1. **Авторизация без форм + телефон в один тап** через `initData` + `requestContact()` → конверсия в бронь х2 vs текущая форма «Перезвоните мне».
2. **CloudStorage для лояльности** → MVP без CRM и БД за пару недель. Карта лояльности, штампы, любимые блюда.
3. **Канал пуш-уведомлений** — бот = бесплатный push для аудитории, согласившейся на `requestWriteAccess`. Без App Store и Apple Push. Идеально для оповещений о смене сезонного меню.
4. **`shareToStory`** → органический трафик, недоступный с обычного сайта.

### Стратегия запуска (рекомендация)

**Фаза 1 (1–2 недели): TMA-обёртка вокруг существующего лендинга.**
- 12 точечных правок к существующему `index.html` (см. §6).
- Создание бота через @BotFather.
- Деплой на тот же Vercel, сабпас `/app` или отдельный проект.
- Тестирование на мобиле.
- **Результат:** МОМО доступен в Telegram, гости получают пуши, бронь через `requestContact` без формы.

**Фаза 2 (4–6 недель): TMA как самостоятельный продукт.**
- Перепроектирование под TMA-навигацию (router, BackButton, MainButton везде).
- CloudStorage для лояльности.
- Сезонное меню с пуш-уведомлениями.
- `shareToStory` на карточках блюд.
- Подключить Restoplace как fallback-канал брони рядом с телефоном.

**Фаза 3 (V3, опционально):** платежи (сертификаты + депозиты банкетов), геймификация, геолокация, биометрия.

### Что НЕ делать

- **Не делать TMA вместо лендинга** — теряем не-Telegram-аудиторию (туристы из РБ/КЗ, поисковый трафик из Яндекса/Google). Лендинг и TMA параллельно.
- **Не делать заказ еды / доставку** — у МОМО нет доставочной инфраструктуры, не наш сценарий.
- **Не использовать tgEDA / Smartofood / Sandyapp как готовое решение** — шаблонный вид понижает воспринимаемое качество. Для семейного ресторана с фирстилем критично сделать кастом или сильно кастомизированную обёртку.

### Экономика

- **Кастомная разработка TMA** (новый продукт, не обёртка): 200–400 К ₽ единоразово.
- **Обёртка существующего лендинга** (Фаза 1): 50–80 К ₽ единоразово.
- **Готовое решение (tgEDA уровень)**: 5–8 К ₽/мес = 60–96 К ₽/год.
- **Точка окупаемости кастома** vs готового решения: ~3 года. Для МОМО кастом оправдан **только если** TMA становится главным каналом коммуникации с местными постоянными гостями (а это реалистично).

### Главное

Для МОМО **TMA — это не «вместо сайта», это «параллельный канал для местных»**. Туристы продолжают находить через Google/Яндекс → лендинг. Местные постоянные (Светлана 52, молодые родители 25–35) → бот в Telegram → TMA → лояльность + пуши о сезонном меню + бронь без формы.

# Экспертная оценка TMA

Дата: 2026-06-05.
Контекст: оценка идеи запустить MOMO как Telegram Mini App с трёх ракурсов. Опирается на §1–§7 исследования выше и на брендовые/контентные решения из [project.md](project.md), [brief.md](brief.md), [design-research.md](design-research.md). Это **отдельная экспертная оценка для TMA**, не путать с прежней «Экспертной оценкой» лендинга (выше в этом файле).

## Эксперт 1: UX-дизайнер (10 лет, специализация на Telegram Mini Apps)

### Ошибки, которые делают новички

1. **«Сайт в рамке»-мышление.** Самая частая ошибка — закинули URL лендинга в `/setmenubutton` и считают, что готово. На выходе: дублирующаяся шапка (Telegram-навбар + sticky-header сайта), cookie-баннер поверх контента, кнопка «Наверх» которой не за что зацепиться, формы с 8 полями. TMA — это **другой контейнер с другими constraints**, не просто веб-вкладка. Минимум **12 точечных правок** (см. §6) даже если HTML/CSS остаётся на 80%.
2. **Свои кнопки вместо `MainButton` / `BackButton`.** Дизайнеры рисуют «красивую стрелку назад» в углу и стандартную кнопку CTA в макете. В TMA это **дублирует** нативные элементы Telegram и съедает экран. Правило: **если действие есть в TMA SDK — использовать SDK**.
3. **Длинные формы.** 8 полей в форме брони (имя, фамилия, телефон, email, дата, время, гости, комментарий, чекбокс ПД, чекбокс рассылки) = **смерть конверсии**. В TMA — **максимум 3 поля**. Остальное либо предзаполнить из `initData`, либо обсудить по звонку.
4. **Свои попапы поверх контента.** TMA сам по себе модал в Telegram. Попап внутри = клаустрофобия. Использовать `WebApp.showPopup()` / `showAlert()` — нативные, выглядят как часть Telegram.
5. **Карусели и слайдеры в основном скролле.** Особенно на мобильном: пользователь скроллит вертикально, попадает на карусель, пытается листать дальше и застревает в горизонтальном скролле. Сетки 2×2 / 1×N всегда лучше карусели.
6. **Не подписываются на `themeChanged`.** Пользователь переключил тему Telegram (светлая → тёмная), но TMA остался светлым. Чёрные цвета бренда на чёрном фоне нечитаемы. Подписка на `Telegram.WebApp.onEvent('themeChanged', updateColors)` — обязательна.
7. **Не убирают cookie-баннер.** В TMA cross-site cookies не работают, баннер не нужен. Но новички тащат его из веба «на всякий случай» — съедает 60px и раздражает.
8. **Тяжёлые JPEG, не WebP.** Webview на Android (особенно Xiaomi/Huawei) медленнее Chrome. Загрузка 1.5 МБ фото = пользователь видит белый экран 3 секунды → закрывает TMA.
9. **iframe-карта Яндекса.** В webview Telegram iframe-виджеты часто не рендерятся или подвисают. Правильно: статичная картинка карты + кнопка «Открыть в Яндекс.Картах» через `openLink`.
10. **Auto-open popups при загрузке.** «Подпишитесь на канал!» / «Хотите получать акции?» — закрывают TMA. Запрашивать согласие **после первого успешного действия** (пример: после успешной брони — «А получать пуши о смене сезонного меню?»).
11. **Не блокируют `disableVerticalSwipes`.** Пользователь скроллит галерею, случайно тянет вниз — TMA закрывается. На страницах с галереями и формами обязательно `WebApp.disableVerticalSwipes()`.
12. **Не используют `HapticFeedback`.** Это «фишка», которая ощущается как нативное мобильное приложение. На каждый тап по кнопке: `HapticFeedback.impactOccurred('light')`. На успех брони: `notificationOccurred('success')`. На ошибку: `notificationOccurred('error')`. Бесплатно, заметный win по «нативности».

### Что ОБЯЗАТЕЛЬНО должно быть

- **`WebApp.ready()` + `WebApp.expand()`** на старте — иначе app открывается в половину экрана.
- **Theme adaptation** через CSS-переменные, маппинг из `themeParams`.
- **`MainButton`** на ключевых конверсионных экранах (Home «Забронировать», Бронь «Отправить»).
- **`BackButton`** на всех не-Home экранах.
- **Safe-area padding-bottom** = `env(safe-area-inset-bottom) + 84px` (под MainButton).
- **`tel:` ссылки** для звонка — работают, нативная звонилка открывается.
- **Body-шрифт ≥ 17px** на mobile — для МОМО критично, primary segment Светлана 52 с пресбиопией.
- **Skeleton при загрузке**, не белый экран.
- **Confirm-before-close** через `WebApp.enableClosingConfirmation()` на страницах с несохранёнными данными формы.
- **`HapticFeedback`** на всех кнопках — нативность.
- **`font-size ≥ 16px` на input** — иначе iOS зумится при фокусе, экран прыгает.

### Что бесит пользователей

1. **Долгая первая отрисовка.** Webview Telegram медленнее браузера. >2 секунды белого экрана = закрытие. Целевой first-paint < 1с.
2. **Шрифт меньше 16px на input** — iOS дёргает зум, контент прыгает, рука сбивается.
3. **Кнопка действия НЕ закреплена внизу** — приходится скроллить вниз. На длинных экранах MainButton обязателен.
4. **Закрытие TMA случайным свайпом** — теряется заполненная форма. Без `disableVerticalSwipes` и `enableClosingConfirmation` это происходит регулярно.
5. **Открытие внешней ссылки в браузере без `openLink`** — пользователь теряет TMA-контекст, возвращается в Telegram через alt-tab.
6. **Тёмный текст на тёмном фоне** — если не подписаться на `themeChanged`, дизайн ломается в тёмной теме Telegram.
7. **Карта Яндекса в iframe — пустой блок** — глюк webview.
8. **Меню как PDF** — на mobile вообще не открывается нормально. Только html-карточки.

### Специфика для МОМО

- Первичный сегмент — **женщины 45–60, постоянные гостьи**. Они **не привыкли к TMA**, для них Telegram — это чаты. Открыли «приложение в Telegram» — это **уже маленькая когнитивная нагрузка**. Поэтому: **минимум экранов** (9–10), **максимум кнопка «Позвонить»** на каждом экране, **никаких хитрых жестов**.
- Контент уже есть в лендинге: 9 секций, 8 подкатегорий кухни. **Переиспользовать**, не дублировать. Адаптация лендинга в TMA — корректное решение (см. §6).
- **Не делать заказ еды / доставку** — у МОМО этого нет, не свой сценарий. Это сильно сократит scope и упростит TMA.

## Эксперт 2: Маркетолог-CRO (специализация на конверсии в TMA)

### Что заставит записаться (превратит просмотр в бронь)

1. **`requestContact` вместо формы — самый сильный win TMA.** Один тап «Поделиться телефоном» → имя из `initData`, телефон от пользователя. **Конверсия в бронь x2–x3** по сравнению с обычной формой. Это **главная причина**, ради которой стоит делать TMA.
2. **Sticky `MainButton` «Забронировать стол».** Всегда виден, всегда в зоне большого пальца. CTR x1.5 vs обычной кнопке в потоке скролла.
3. **Промо-баннер на Home «Сезонное меню Севастопольские сезоны до 31 мая»** — ситуативное действие. Гость видит ограничение по времени → больше шанс действовать сейчас.
4. **Социальный сигнал в Hero:** «★ 4.9 · 2045 отзывов · с 2013 года · средний чек 1500 ₽» — снимает страх первой брони. Так же работает, как и в вебе.
5. **Микрокопия под MainButton:** «Звонок бесплатный · Перезвоним за 5 минут» — снимает скрытый страх «опять обзвон-спам».
6. **Карта лояльности «штампы»** в `CloudStorage` — для постоянных. После 5 посещений — бесплатный десерт. **МОМО — идеальный кейс**: 60% выручки с постоянных, лояльность критична.
7. **Пуш «Новый сезонный сет уже в зале»** через бота — возврат через 1–2 месяца, без затрат на рекламу.
8. **`shareToStory` на карточке блюда** — гость публикует «Я в МОМО» с фото в Telegram-сторис. **Органический трафик**, недоступный с веба.
9. **`HapticFeedback.notificationOccurred('success')` после брони** + **`WebApp.close()`** → гость возвращается в чат и видит подтверждение от бота. **Чувство завершённости** = удовлетворение = повторное обращение.

### Что повышает конверсию (специфика TMA)

| Инструмент | Win | Размер эффекта |
|---|---|---|
| `initDataUnsafe.user` для предзаполнения имени | Не вводят имя — меньше отказов | +10–15% completion |
| `requestContact()` вместо ручного ввода телефона | Меньше отказов на финальном шаге | +30–50% completion |
| `MainButton` вместо inline-кнопки | Всегда в фокусе на mobile | +15% CTR |
| `WebApp.close()` после успеха | Чувство завершённости, не lost context | +10% NPS |
| `addToHomeScreen` для постоянных | Иконка МОМО на главном экране = PWA-аналог | +30% retention |
| `CloudStorage` для «повторить прошлую бронь» | 1–2 тапа vs полный flow | +40% repeat orders |
| Пуш о смене сезонного меню | Холодный re-engagement через бот | +15–25% return visits |
| `shareToStory` | UGC, органика | +5–10% к трафику в сезон |

### Антипаттерны конверсии

1. **Длинные формы** — каждое лишнее поле срезает 5–10% completion. Максимум 3 поля.
2. **Просьба на push в первую секунду** — отказ. Запрашивать `requestWriteAccess` **после первого успешного действия** (бронь, заказ сертификата).
3. **Auto-open popup «подпишитесь на канал»** — закрывают TMA немедленно.
4. **Цена «по запросу»** — лишает доверия, как и в вебе.
5. **5+ CTA на одном экране** — рассеивает фокус. Один главный (`MainButton`), 1–2 второстепенных (текст-ссылки).
6. **Просить email отдельно** — у TMA есть `requestContact`, email = лишний канал, не работает в РФ.
7. **«Сделать заказ на доставку»** для МОМО — это **не сценарий**. Если поставить эту кнопку — пользователь её жмёт, узнаёт, что доставки нет → возмущение. Уберите этот сценарий из карты экранов вообще.
8. **«Зарегистрироваться»** — у пользователя уже есть Telegram-аккаунт. `initData` решает всё.

### Сегментная стратегия (3 первичных сегмента МОМО)

| Сегмент | Главная фича TMA | Маркетинговый канал |
|---|---|---|
| **Светлана 52, постоянная** | Карта лояльности + пуш о смене сезона + повтор прошлой брони | **ЦЕНТР TMA-стратегии**: bot-канал, штамп-карта, персональные предложения через CloudStorage |

### Воронка TMA (4 шага)

```
1. ОТКРЫЛ ────────► Home за 1 сек: понял где, что есть, MainButton «Забронировать»
     ▼ обрыв 20%        метрика: bounce < 30% (better than web!)

     ▼ обрыв 10%        метрика: CTR > 20%
```

## Эксперт 3: Telegram developer (4 года, делал 20+ TMA)

### Ограничения, о которых надо знать

1. **TMA только в HTTPS.** Никакого http. Vercel/Cloudflare Pages — ok.
2. **Mobile-first.** Telegram Desktop открывает TMA, но UX обычно странный. Полагайся на mobile-тесты.
3. **Webview-движки:**
   - iOS — WKWebView (Safari-engine)
   - Android — WebView (Chrome-engine)
   - Разница в производительности существенная. Тестируй на обоих.
4. **`CloudStorage` лимиты:** 1024 ключа × 4096 байт. **Не для больших данных**, не для бинарников. Карта лояльности (10 ключей × 100 байт) — отлично. История заказов с фото — нет.
5. **Cookies в webview ненадёжны.** Третьеперсонные cookies блокируются. Используй `CloudStorage` или localStorage для важного.
6. **`addToHomeScreen` (8.0)** — работает не на всех устройствах. iOS требует helper-page, Android более или менее ok. **Не критичная фича**.
7. **`HapticFeedback`** — на iOS отлично, на Android частично (зависит от устройства). Тихий fallback приемлем.
8. **No native push.** Push идёт через бота как обычное сообщение → нужно `requestWriteAccess` от пользователя. Без него бот не может писать первым.
9. **Bot Payments в РФ 2026 — главный pain.** ЮKassa — единственный условно стабильный провайдер. Фискализация по 54-ФЗ требует своей интеграции (выйти из Telegram-шита, открывать `openLink` на ЮKassa-страницу). Telegram Stars — только цифровые товары, для сертификатов ресторана серая зона.
10. **No Service Workers.** TMA — не PWA, offline-first невозможен на 100%. Хотя бы skeleton при загрузке + cache важных ассетов через HTTP-кэш.
11. **BackButton vs iOS-свайп.** iOS-пользователь свайпает справа налево чтобы закрыть → это **закрывает весь TMA**, а не возвращает на предыдущий экран. `enableClosingConfirmation()` спасает только частично (показывает alert, но не возвращает в state). На страницах с формами — обязательно.
12. **`viewportChanged`.** Клавиатура открывается, viewport уменьшается. Контент съезжает, MainButton прыгает. Подпишись на event, пересчитай layout.
13. **iframe Яндекс.Карт глючит.** Подтверждённый баг webview. Только статичная картинка + `openLink`.
14. **Google Fonts периодически блокируются.** Особенно на Android в РФ. Self-host шрифтов в `/fonts/` или system-fonts (-apple-system, Segoe UI, Roboto).
15. **Версионирование API.** `WebApp.isVersionAtLeast('8.0')` ОБЯЗАТЕЛЬНО перед использованием новых методов. Старые клиенты Telegram молча игнорируют — без проверки баг непонятный.

### Что работает хорошо (надёжные паттерны)

- **`tel:` ссылки** — нативная звонилка, кросс-платформенно.
- **`WebApp.openLink(url)`** — внешние ссылки в системный браузер. Стабильно.
- **`WebApp.openTelegramLink(url)`** — внутренние Telegram-ссылки (`t.me/...`). Стабильно.
- **`initData` HMAC валидация** — стандарт, документировано, надёжно. Не пиши свою авторизацию.
- **Theme adaptation** через `themeParams` + `themeChanged` event — работает на 100% устройств.
- **`MainButton` / `BackButton`** — стабильно, поддержка с 6.1.
- **`HapticFeedback` на iOS** — отличный, мгновенный.
- **`shareToStory`** — работает, но требует медиа на CDN (не локальные blob:).
- **`openInvoice` через ЮKassa** — для тех, кто прошёл фискализацию.

### Что лучше НЕ делать

1. **Свой login / OAuth** — есть `initData`, всё.
2. **PWA-фичи внутри TMA** — конфликт. Service workers, manifest.json — не нужны.
3. **iframe внешних виджетов** — Яндекс.Карты, YouTube — глюки. Только `openLink`.
4. **Тяжёлые JS frameworks без tree-shaking** — Next.js (App Router) с SSR в TMA = overhead. Vite + React/Solid + code-split = правильно.
5. **WebGL / 3D-сцены** — лагают в webview.
6. **Тяжёлые анимации, parallax** — тормозят на средних Android.
7. **Обход Bot Payments для физических товаров** — Telegram банит.
8. **Игнорировать `viewportChanged`** — клавиатура ломает layout.
9. **localStorage для критичных данных** — webview сбрасывает иногда. Используй `CloudStorage`.
10. **Не использовать `disableVerticalSwipes`** на страницах с формами — пользователь случайно свайпом закроет, потеряет данные.

### Tooling и инфраструктура

- **Локальная разработка**: ngrok / cloudflared для HTTPS. Без HTTPS Telegram не откроет TMA даже на localhost.
- **Стек MVP**: Vite + React + `@telegram-apps/sdk-react`. Стартер: `npm create @telegram-apps/mini-app@latest`.
- **Хостинг**: Vercel / Cloudflare Pages. У МОМО уже Vercel — можно тот же проект, сабпас `/app`.
- **Bot**: grammY (TS, Node) или aiogram 3.x (Python). На Cloudflare Workers + D1 (SQLite) бесплатный план тянет до ~10К пользователей.
- **Deep links**: `https://t.me/momo_sev_bot/app?startapp=banquet` — параметр читается через `WebApp.initDataUnsafe.start_param`. Для QR-кодов в зале.
- **Аналитика**: `@telegram-apps/analytics` (официальный) или свой эндпоинт + Plausible. Google Analytics / Метрика часто глючат в webview.

### Главные риски для МОМО

1. **Bot Payments в РФ нестабильны** → откладываем платежи в V2/V3, MVP без них.
2. **Пуш через бота требует `requestWriteAccess`** — не все согласятся. Прогнозируемая база подписок: 30–40% от пользователей TMA.
3. **CloudStorage лимиты** → лояльность ок, история заказов с фото — нет.
4. **Зависимость от Telegram-инфраструктуры** — если завтра Telegram заблокируют в РФ окончательно, TMA умирает. **Поэтому критично: лендинг остаётся главным каналом**, TMA — параллельный.
